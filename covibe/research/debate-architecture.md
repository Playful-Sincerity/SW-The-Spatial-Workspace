## CORE THESIS

The server approach solves a problem CoVibe doesn't have in Phase 1 — production-grade real-time infrastructure — while creating problems it does: deployment friction, security architecture decisions, and the risk of spending a week building server plumbing when the actual unknown is whether cross-session Claude insights are useful at all. The git-based approach validates the core concept today, with the tools already running, without writing a line of new infrastructure code. Prove the insight is worth having before engineering the pipe it travels through.

## KEY ARGUMENTS

1. **Phase 1's true goal is concept validation, not production feel.** CoVibe's CLAUDE.md Phase 0 uses shared VM + tmux and VS Code Live Share — deliberately crude tools — to "learn what works and what's missing." Phase 1 should answer: "Does having cross-session Claude awareness actually change how Wisdom and Frank work?" A git-based system answers that question. If the answer is "yes, but the 10-second delay kills it," then Phase 2 can be the server. If the answer is "actually the insights aren't that useful" or "we need something fundamentally different," then no server infrastructure was wasted. The MVP principle is brutal here: build the minimum that tests the hypothesis, not the minimum that feels good in a demo.

2. **The git approach works today with zero new code.** Every Claude Code session already runs git. A shared GitHub repo, a cron-triggered git push every 5 seconds, and a git pull on the other end requires no new dependencies, no deployment, no server maintenance, no Railway account, no security architecture decisions. The CON position's strongest asset is operational simplicity: if Wisdom's Railway server goes down at 2am during a demo, the session is dead. If GitHub goes down, the session is also dead — but GitHub's uptime SLA is better than a personal Railway deployment. Operational risk is real for Phase 1 demos.

3. **The latency requirement is asserted, not demonstrated.** The PRO position claims sub-second latency is required for the "minimum viable magic." But this is an assumption about user experience that hasn't been tested. Human thought takes 5–30 seconds to develop into a coherent prompt. Claude's response takes 15–60 seconds to generate. In that cycle, a 5–10 second synchronization delay is within the natural rhythm of the collaboration — it's not a UX blocker. The `sshx` comparison is a strawman: sshx requires real-time because it's a shared terminal where keystrokes must arrive immediately. CoVibe is a higher-level, message-based interface where latency is measured in seconds, not milliseconds. The "feel" difference between 2-second and 10-second message delivery in this use case is marginal, not categorical.

4. **The server approach front-loads the hardest security decision.** The component scout notes it clearly: "The Agent SDK process has full Bash, file system, and network access. In production, it must run in an isolated container." This is a non-trivial security architecture choice that must be made before the server goes live. File-based git messaging sidesteps this entirely: each session runs locally on the participant's own machine, with their own filesystem isolation. Git is the only shared surface, and GitHub's security model is already solved. The server approach requires deciding on containerization (Docker? E2B? Fly.io?) and implementing it before Phase 1 is actually safe to demo to anyone other than Wisdom and Frank on a trusted network.

5. **Git is the natural collaboration substrate for a coding tool.** Claude Code is fundamentally a git-native tool — sessions write to `.claude/projects/`, changes are committed, context accumulates in version history. File-based cross-session messaging isn't fighting the grain of the tool; it's extending the git substrate CoVibe already depends on. The PRO position treats git as a legacy workaround; the CON position treats it as the substrate. The CoVibe CLAUDE.md lists "git integration (commit, branch, PR from the UI)" as a Phase 2 feature — meaning Phase 2 deepens the git relationship, not replaces it. Starting with git-based messaging is Phase 1 leaning into where Phase 2 goes anyway.

## PREEMPTIVE REBUTTAL

The strongest counter-argument is: "claude-agent-kit already solves the server complexity — SessionManager and WebSocketHandler are built, it's 150 lines of adaptation." Granted — claude-agent-kit dramatically reduces the server implementation burden. But "150 lines of adaptation" still means: understand the claude-agent-kit API, write the SharedSession routing layer (~150 lines), write the TurnManager (~100 lines), write the room code auth (~20 lines), configure Railway deployment, set up environment variables, decide on containerization. The component scout's own estimate is "~1 week of focused building." Compare that to git-based: one shell script (auto-push every 5 seconds), one Claude session that watches the shared repo, and a shared GitHub repo. That's an afternoon. The question isn't whether the server is buildable; it's whether the week spent building it is the right bet for a Phase 1 whose job is to validate the concept, not ship a product.

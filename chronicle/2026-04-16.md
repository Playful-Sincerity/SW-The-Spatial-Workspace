## 04:48 — Implementation

**What:** Built `generator/watch-server.py` — a stdlib-only HTTP server that serves the canvas, polls the filesystem, regenerates on debounce, and exposes `/__snapshot.json` for client-side live-reload.
**Why:** "Constantly updating to see our ecosystem" was a core ask. File-watch + auto-regen + auto-reload-on-hash-change covers the full loop.
**Means:**
- Polls every N seconds (default 5) using `os.walk` with the same exclusions as the generator.
- Signature is sha256 of `path|mtime|size` for every `.md` file. Cheap, deterministic, no inotify dependency.
- When signature changes → run generator subprocess → recompute output hash → `/__snapshot.json` returns the new hash.
- Client polls `/__snapshot.json` every 2s. When the hash changes, soft-reload the page.
- Header dot pulses green when live, turns orange during regen.
- All stdlib. No fswatch, no Node, no install step.
**Files:** `generator/watch-server.py`

## 05:02 — Decision

**What:** Used `d3.cluster()` (all leaves at perimeter) instead of `d3.tree()` (depth-proportional radius) for radial layout.
**Why:** The ecosystem's depth varies wildly (2 to 8+ levels). With `d3.tree()`, dense subtrees crowd the center while shallow ones stretch out. With `d3.cluster()`, every leaf — file or folder — lands on the same ring, making the umbrella structure read clearly.
**Means:** Trade-off accepted: depth is encoded by *which ring an internal node sits on*, not by absolute distance. The visual cost is negligible; the legibility win is large.

### Fix #3 — Overlap / physics rebalance

**Diagnosis:** Quadtree-backed `rectCollide` had two real problems.

1. **Quadtree cull bounds were too tight.** The visit-callback culled at `a.x ± maxR` where `maxR` is only `a`'s own half-extent. But d3.quadtree indexes node centers, not rectangles. A wide neighbour `b` whose center is outside `a`'s bbox but whose far edge still extends into `a`'s collision zone was silently skipped. Classic "rectangles in a point-quadtree" bug. **Fix:** track the max width/height across all nodes at `force.initialize` time, then expand the visit cull by `maxOther/2` in each direction.

2. **Force imbalance.** `forceRadial` strength 0.22 at depth 1, decaying to 0.08 at depth 5, was strong enough to re-pin nodes to their nominal rings faster than collision could push them off. When 400 leaves want to sit on a ring whose circumference can only fit 100 at their label widths, collision can only push tangentially along that ring — they just stack. **Fix:** radial strength now 0.14 / 0.08 / 0.02 by depth; velocityDecay up from default 0.4 → 0.55 for more decisive settling; collision strength 1.4 → 3.0; padding 10 → 14; iteration budget 300 → 500 base; density scale cap 2.2 → 2.8.

### Diagnosis

Also a minor bug: the crowding distance formula in `seedLocal` uses the parent's `_nominalWidth` as the child-width estimate, which is wrong — children are usually smaller than parents. Produces over-estimated seed distances. Physics would correct this if link force wasn't already stretching things.

### Proposed next (not executed — awaiting Wisdom's go-ahead)

Strip the layout back to a single coordinate system:

- **Delete** the global angular `allocate()` function — unused after the local seed.
- **Delete** `radiusAt[]` and everything that computes it.

- **Rewrite** the physics link distance to `parentHalf + childHalf + gap` — purely local, no dependency on any global ring scheme.
- **Fix** the crowding formula in `seedLocal` to use a fixed child-width estimate (~120 px) or the actual child's nominal width.

**Why:** The old layout had four interfering coordinate systems (global angular allocation, global ring radii, local per-subtree seed, physics polish). Deep subtrees got flung thousands of pixels from parents because physics link distance read from radiusAt[] which was ~10,000px at deep depths.
**Means:** The entire layout is now: computeNominalWidths → computeLeafCounts → computeSubtreeExtents → applyRootGrowth → seedLocalFan → polishPhysics (link + charge, NO radial) → resolveOverlaps (Jacobi collision solver). One coordinate system throughout.

**Architecture decisions made during implementation:**
1. **Collision as post-process, not inline force.** rectCollide modifies positions directly, but d3-force's velocity integration undoes those corrections in the same tick. Moved collision to a dedicated Jacobi constraint solver (`resolveOverlaps`) that runs AFTER the physics sim settles. The Jacobi approach accumulates all displacements per node before applying them (vs. Gauss-Seidel's immediate application which causes oscillation).
2. **Subtree-aware fan distances.** The brief's algorithm used child nominal widths for fan distance. At 3000+ nodes, this placed subtrees too close together. Added `computeSubtreeExtents` using sqrt(leafCount) × nodeWidth — grows sublinearly so it doesn't compound exponentially like a recursive formula would.
3. **Root growth via _fontScale.** Root gets 2.0× growth on width, height, and _fontScale. The render code reads _fontScale to scale the font proportionally.

**Overlap results:**
- Cold-open (4 nodes): 0 overlaps ✓
- Branch-expand (3078 nodes): 437 overlaps — NOT zero
- Full-expand (5504 nodes): ~739 overlaps — NOT zero
- Overlaps are concentrated in deeply nested directories with many long-named files (session briefs, people profiles)

**What's NOT at zero and why:** The Jacobi solver converges to ~400-450 overlaps on the branch-expand case despite 1000 passes. The remaining overlaps are in cascade chains where pushing node A away from B pushes A into C, and correcting that pushes C into D. The double-counted push (each pair processed from both sides in the quadtree scan) provides aggressive convergence but hits an oscillation floor. Getting to true zero likely requires either (a) a smarter seed for high-child-count directories (spiral/grid packing for 50+ siblings), or (b) a constraint solver that tracks which pairs are oscillating and dampens them selectively.
